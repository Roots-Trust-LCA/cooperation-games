What It Is

Parachute is building open infrastructure for AI memory — the layer between humans and AI systems that holds structured, persistent knowledge. The core thesis: conversation history is not memory. Notes, tags, links, and relationships in a traversable graph are memory. Parachute gives any AI (Claude, ChatGPT, Gemini) access to that graph via the Model Context Protocol (MCP), without vendor lock-in.

The project is organized as Open Parachute, PBC — a Colorado Public Benefit Corporation. This is a structural sibling to RegenHub, LCA: both operate from Colorado, both are mission-driven entities in the cooperative/public-benefit space.

Design stance: Local-first, AGPL-licensed, single-tenant-by-default. Data lives in SQLite on your machine. The hosted cloud offering wraps the same code. Third parties can build modules under any namespace — no central registry required.

Module Protocol — The Composability Layer

The most architecturally significant piece is the module protocol — a set of canonical HTTP contracts that let any service plug into the Parachute hub:

EndpointPurpose
GET /.parachute/infoModule identity — name, version, kind, capabilities. No auth, CORS *.
GET /.parachute/icon.svgInline SVG icon for hub display.
GET /.parachute/config/schemaJSON Schema of configurable properties (Phase 2+).
GET /.parachute/configCurrent config values (requires admin scope).
GET /.well-known/parachute.jsonService manifest — declared scopes, dependencies, endpoints, startCmd.

Modules declare dependencies and the hub auto-wires them. A module that needs Scribe declares it; when both are installed, the hub injects Scribe's URL into Scribe's config automatically. The ambition is a self-assembling ecosystem — add a module, capabilities appear.

Identity Layer — Hub as OAuth Issuer

The hub is the identity authority. OAuth 2.1 + PKCE. Services are OAuth resource servers, not issuers — they validate hub-issued tokens.

  • Scope model: vault:read, vault:write, scribe:transcribe, hub:admin, channel:send. Hierarchical — vault:write implies vault:read.
  • Service catalog in token response: OAuth token carries the complete service directory (vault URL, scribe URL, versions). Clients configure themselves from one auth exchange.
  • Phased: Phase 0 (today) — hub proxies to Vault's auth. Phase 1 — token carries service catalog. Phase 2 — real scope enforcement. Phase 3 — auth extracted to dedicated service.
  • Multi-vault scopes: Planned as vault:<name>:read format. Not yet implemented.

Cloud Offering

Parachute Cloud wraps the self-hosted vault in hosted multi-tenant infrastructure. Architecture: single Cloudflare Worker dispatcher → one Durable Object per vault (fully isolated SQLite) → R2 for attachments → D1 for account/subscription metadata.

Enterprise & Organizational Gap Analysis

Where the product is today vs. what an LCA instance would require.

CapabilityCurrent StateGap for LCA Use
Multi-user vault Single-owner design. One password, one primary user granting consent to AI clients. Members need individual identities within a shared organizational vault. Role-based access (steward, member, observer).
Team knowledge base Multi-vault on one server (separate isolation per vault). No shared editing. Org-level vault with member notes, agreements, decisions, patronage records accessible to authorized members + agents.
SSO / federated identity Owner password + TOTP only. No external IdP. LCA members should authenticate via existing identity (email, wallet address, Farcaster ID).
Note-level ACL Vault-level isolation only. Within a vault, all-or-nothing for a given token. Private member notes vs. collective agreements vs. public-facing content — distinct permission tiers.
Agent access control Bearer tokens with full/read scopes. Nou and Dianoia need read/write on specific vaults; guest agents (future) need sandboxed read-only.
Structured data schemas Notes are freeform text + tags + links. No enforced schema on note type. Member records, agreements, capital account entries, patronage data need typed fields — REA ontology alignment.
Audit trail / immutability Standard CRUD. Backup/retention for recovery, not audit. LCA governance requires durable, verifiable record of decisions and capital account changes.
Webhook → external systems Webhook triggers for Scribe/Narrate. Generic pattern exists. Hooks into Supabase (CIS), Workshop coordination surface, on-chain settlement. Pattern is right; needs org-scoped triggers.

What's Strong

MCP as the lingua franca. Building on MCP is the correct bet. Any agent that speaks MCP — Claude Code, future models — gets vault access for free. The ecosystem doesn't depend on one AI provider surviving.
Module protocol is genuinely extensible. The /.parachute/* contract set is clean. Third-party modules don't need a central registry. An LCA-specific module (CIS integration, patronage accounting read layer) could be built as a standard Parachute module.
Cloudflare-native architecture for cloud. Durable Objects per vault is the right isolation model for multi-tenant. Avoids shared-state contention. Geographically distributed automatically. Edge-native from day one.
AGPL + Colorado PBC. Values alignment with RegenHub, LCA. Open source as trust. Public benefit chartered. Both entities emerging in the same state, same era, adjacent to cooperative / solidarity economy circles.
Voice-native path. Parachute Daily + Scribe + Narrate form a voice-first journaling loop. For an LCA capturing meeting knowledge, voice transcription → structured vault note → agent-readable context is a high-value workflow.

Where It's Early

Cloud offering is scaffold only. The architecture is well-designed but not deployed. No enterprise tier, no team vaults, no organizational pricing. This is the obvious next frontier.
Single-tenant by design, multi-tenant by exception. The core vault assumes one owner. Multi-user workflows require either separate vaults per person (with federation TBD) or upstream work on note-level ACL.
Agent framework (parachute-agents) is very early. The pattern is right — markdown-defined agents with Vault integration. But it's pre-production. Stateful agents on Cloudflare Durable Objects are the right long-term shape.

Relevance to Techne Studio / RegenHub, LCA

Immediate LCA use cases for a Parachute instance

  • Member knowledge base: Organizer notes, agreements, decisions, governance records — structured in a vault, queryable by Nou via MCP.
  • Meeting capture pipeline: Voice recording → Scribe → structured note → tagged in graph → Nou reads context for next meeting. Replaces ad-hoc notes in scattered docs.
  • Agent context persistence: Nou and Dianoia's working memory across sessions. Currently in flat workspace files; a vault gives structured, linkable, searchable context.
  • Common Information System bridge: Vault webhooks → Supabase CIS. Member records in Parachute, financial records in CIS, synchronized via agent-triggered hooks.
  • Patronage documentation: Each member's patronage log as vault notes with structured metadata. Agent-readable for audit and reporting.

Where product research from RegenHub could shape enterprise Parachute

  • Multi-member vault model: Define what org-level vault looks like — member identity, role-based scopes, shared vs. private namespaces. LCA structure is a concrete test case.
  • Structured note schemas: Agreements, capital account entries, governance votes — typed note schemas with required fields. Drives the vault toward REA ontology compatibility.
  • Agent access tiers: What scopes should an org's agent (Nou) have vs. a member's personal agent vs. a guest/external agent? Defines the enterprise permission model.
  • Webhook → external LLM systems: Parachute → Workshop coordination surface. Meeting note created → sprint proposed automatically. Concrete integration pattern for enterprise workflows.
  • Cross-org federation: Two LCAs sharing selected vault content. Defines the federation model for Parachute in cooperative networks.

Technical Advisor Framing

The natural advisory focus is enterprise / organizational product research — using the LCA's operational needs as a living specification. This is valuable to Parachute because:

  • The LCA's governance, patronage accounting, and multi-agent coordination requirements are harder than typical enterprise SaaS needs. A cooperative with AI agents is an edge case that reveals real architectural limits.
  • Techne Studio's work (CIS, Workshop, cooperation.games) produces concrete integration requirements — not hypothetical enterprise features, but real workflows needing vault support.
  • RegenHub's network (Kevin Owocki, Gitcoin / regen economy adjacency) connects Parachute to the cooperative / solidarity economy space — a distribution channel for values-aligned enterprise adoption.
Proposed research focus areas:
  • Org-level vault architecture — multi-member, role-scoped, federation-ready
  • Structured note schemas and REA ontology compatibility
  • Agent access tiers for organizational deployments
  • Webhook integration patterns for external coordination surfaces
  • Pricing and packaging for cooperatives / small orgs (between Personal+ and Pro)
